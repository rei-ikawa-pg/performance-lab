<context>
あなたは「Performance Lab」というWebアプリケーションを開発します。
任意のURLを入力するとPageSpeed Insights API経由でCore Web Vitals（LCP, CLS, INP, FCP, TTFB）を計測し、スコア可視化・改善提案レポートを提供するツールです。

技術スタック:
- Next.js 15 (App Router) + TypeScript 5.x
- Tailwind CSS + shadcn/ui
- D3.js（ゲージチャート。実装困難な場合はRechartsにフォールバック）
- TanStack Query（データフェッチ・キャッシュ管理）
- NextAuth.js v5（Google OAuth認証）
- Supabase（PostgreSQL — 計測履歴保存）
- Vitest + React Testing Library + Playwright
- デプロイ先: Vercel

プロジェクトのディレクトリ構成:
performance-lab/
├── src/
│   ├── app/

<instructions>
docs/02_実装タスク.md に記載されたタスクを Phase 単位で順に実装してください。
各 Phase の作業開始前に、該当タスクの一覧を確認し、完了したタスクにはチェックを入れて docs/02_実装タスク.md を更新してください。

実装時の原則:
1. Server Components をデフォルトとし、インタラクティブな部分のみ "use client" を付与する
2. APIレスポンスは必ず型定義し、zodでランタイムバリデーションを行う
3. PageSpeed Insights APIキーは process.env.PAGESPEED_API_KEY でサーバーサイドのみ参照する
4. スコア判定は Google 公式基準に準拠する:
   - LCP: Good ≤2500ms, NI ≤4000ms, Poor >4000ms
   - CLS: Good ≤0.1, NI ≤0.25, Poor >0.25
   - INP: Good ≤200ms, NI ≤500ms, Poor >500ms
   - FCP: Good ≤1800ms, NI ≤3000ms, Poor >3000ms
   - TTFB: Good ≤800ms, NI ≤1800ms, Poor >1800ms
   - カラー: Good=#0CCE6B, NI=#FFA400, Poor=#FF4E42
5. D3.js は dynamic import で遅延ロードし、バンドルサイズに注意する
6. Rate Limiting は /api/measure に対して 1ユーザー10リクエスト/時で実装する
7. RLS ポリシーにより、ユーザーは自分のデータのみ参照可能とする
8. アクセシビリティ: チャートに aria-label、カラーコントラスト比 4.5:1 以上、キーボード操作対応
9. テストは以下を優先する:
   - ユーティリティ関数・APIルート: ユニットテスト（カバレッジ80%以上）
   - 計測フロー: Playwright E2Eテスト
10. 不明な仕様・判断に迷う点があれば、実装前に確認してください

Phase の進め方:
- Phase 1 → Phase 2 → Phase 3 → Phase 4 の順に実装する
- 各 Phase 完了時に、動作確認可能な状態にする
- D3.js ゲージチャートが Phase 1 の PoC で困難と判断した場合、Recharts にフォールバックし、その旨を報告する
- Phase 2 終了時点で Must 機能が未完の場合、Phase 3 の Supabase 連携をスキップして Must 完成を優先する
</instructions>

<constraints>

- 環境変数に実際の値をハードコードしない

- 不明な場合は推測せず確認する
</constraints>
